# 语音交互与情绪识别架构设计 (Voice Interaction & Emotion Recognition Design)

## 一、背景与目标 (Overview)

为了打破传统文字 AI 伴侣的交互局限，本项目致力于实现 **全双工语音交互 (Voice-to-Voice)**。核心目标不仅在于实现“语音转文字”的工具化，更在于提取音频中隐含的**情绪与语气特征**，使底层大语言模型（LLM）能感知用户的非文本意图，并生成更具情感共鸣的回复。

## 二、ASR (语音识别) 引擎选型

在 Telegram 环境下，用户发送的语音消息通常采用 Ogg/Opus 格式。为了实现毫秒级流式响应，我们对多个 ASR 引擎进行了评估：

### 方案 A：Groq (基于 Whisper-large-v3) —— [最终采纳]
- **优势**：依托专有硬件加速，推理速度极快，能够在用户发送语音后近乎瞬时返回文本，大幅提升了交互的实时性。
- **架构集成**：
  1. 通过 `ffmpeg` 拦截 Telegram 发送的 Ogg/Opus 音频流并进行重采样。
  2. 将重采样后的音频片段发送至 Groq ASR API。
  3. 提取转录文本。

### 方案 B：OpenAI Whisper
- **优势**：标准 API 接入，多语种支持稳定。
- **劣势**：相较于 Groq 方案，在响应速度上略有逊色，且成本较高。

> **结论**：系统默认集成 Groq 方案作为首选，以追求极致的对话流畅度。

## 三、SER (语音情绪识别) 方案设计

在解决了“听懂”之后，系统需要“感知”用户的情绪。我们设计了两种维度的架构：

### 1. 基于多模态融合的文本隐式情绪提取 (Text-Based Pragmatic Approach)
- **机制**：在 ASR 转录完成后，并非直接将文本发送给主线角色扮演模型，而是引入一个前置的微型“情绪嗅探 Agent”。
- **实现逻辑**：
  利用 LLM 的意图识别能力，使用定制的 Prompt（例如：“分析当前文本中隐含的语气，如愤怒、开心、失落、急促”），提取情绪标签。
- **上下文拼接**：
  提取出的情绪标签作为隐式的 Context (例如 `*(用户正带着愤怒的语气说道)*`)，自动注入到主线对话流中。
- **优势**：零额外架构成本，复用现有的大语言模型推理能力，能处理绝大部分常规情感场景。

### 2. 基于纯声学特征的物理提取 (Audio-Based Advanced Approach)
- **机制**：引入专用的音频情绪分析模型（如 Hume AI Expression Measurement）。
- **实现逻辑**：直接分析音频的声调、语速、甚至笑声和叹息，输出高维情绪向量。
- **优势**：能够捕捉到超越文本本身的纯物理音频特征（如讽刺、尖叫）。

> **当前部署状态**：由于 API 依赖限制，当前版本（MVP）主要采用 **Approach 1**，并在代码结构上保留了向 Approach 2 平滑升级的接口抽象。

## 四、全链路语音数据流 (Data Flow)

1. **输入捕获**：Telegram Adapter 监听到 `filters.VOICE` 事件，自动下载临时语音文件。
2. **格式转换**：调用底层 `ffmpeg` 与 `pydub` 将文件清洗并转码为标准采样率格式。
3. **识别与情绪嗅探**：
   - ASR 模块 (`asr_client.py`) 返回文本。
   - 情绪嗅探模块提取情感隐变量。
4. **角色推理**：文本 + 情绪标签 传入大模型进行角色扮演推理。
5. **TTS 渲染**：将模型生成的纯文本（过滤掉内心独白和动作描写 `*...*`）发送至 TTS 引擎（Deepgram Aura / OpenAI），根据角色性格进行高保真发音。
6. **语音下发**：将合成的音频转化为 Telegram Voice 格式发送给用户，完成一次完整的闭环。
